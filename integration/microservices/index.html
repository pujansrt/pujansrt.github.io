Microservices Explained

Monolithic Architecture

In monolithic architectural design we create a big cumbersome app with all modules tightly coupled inside a single ear or war like apps, which is typically deployed on a web or app server. This encounters multiple problems here:

  • Development Plan - As monolithic apps grow in size, due to tight coupling between components, it becomes difficult to do easy and frequent releases.
  • Continuous delivery - If a single change to the app would require the entire app to be redeployed, then this could become an obstacle to frequent deployments, and thus an impediment to continuous delivery. This could be a serious issue if you are serving a mobile app where users expect the latest cool new features all the time.
  • Difficult to manage team and project - Project management has its own challenges in monolithic app development. Even a modularized app has interdependency in terms of deployment and release. It takes a toll in terms of time and effort to plan the release and manage tightly coupled interdependent modular development.
  • Expensive Scalability - It is possible to scale a monolithic app but the cost is very high.
  • Lack of technology diversity - We have to choose a balanced stack which can serve well for all of our requirements. We can not employ specific technology for specialized needs.
  • Component Replacement - It is very difficult to replace any component with a better design or performance without affecting the whole architecture.

Microservices Architecture

In Microservices architecture, components are independent of each other. These components are mini apps, which are developed separately. They will follow their own development and deployment lifecycle. In a general setup we can have scenarios where we need data from various components for a single request. Ideally, it should have a APIs or FrontController which aggregate data from these components and give response back. These components should talk to each other via various methods such as REST APIs, EJBs, WebServices, JMS (message queue) etc. There are some key characteristics to write here:

  • Independently running components classified around some business capabilities.
  • It has product mentality over project.
  • Decentralize standards and Decentralized data management.
  • Automated infrastructure management. For deployment of independent components, we need to rely on automated infrastructure management to reduce the complexity.
  • Evolutionary design for getting the best possible decomposed system, which can be replaced and upgraded without affecting its collaborator.
