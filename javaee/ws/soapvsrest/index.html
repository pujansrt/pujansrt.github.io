SOAP vs REST

Basic Concepts

REST

  • Makes data available as a resources. E.g. AccountInformation, Invoice
  • Makes application logic available as a servies. E.g. getAccountInformation, PayInvoice

SOAP

  • It’s an architectural style.
  • No strict contract between client and server.
  • It’s a protocol.

Advantages

REST

  • It uses standard HTTP. Easy to implement.
  • Returns data in many different formats (JSON, XML, etc.)
  • Better performance and scalability. Reads can be cached.
  • REST allows better support for browser clients due to its support for JSON.

SOAP

  • Works on top of any communication protocol.
  • It can be fully described using WSDL.
  • Provides end­to­end reliability and successful/retry logic is built in.
  • Security and authorization are part of the protocol.

Disadvantages

REST

  • Only works on top of HTTP protocol.
  • No built­in standards for security or reliability.
  • No constraints on the payload.

SOAP

  • Hard to implement and not so popular among web and mobile developers.
  • Permits XML data format only.
  • SOAP defines standards to be strictly followed.
  • Requires more bandwidth and resource than REST.

When to Use What

REST

  • When your bandwidth is very limited.
  • When client and server operates on a web environment.
  • Examples: Social Media Service, Web chat service

SOAP

  • When performing transactions that involve multiple calls.
  • When you want to enforce a strict contract between client and server
  • Examples: Financial services, telecommunication services

In nutshell, identify your project need and goal you want to achieve before you pick any of the approaches.
